# CedarGuard — Architecture

> One product, multiple deployment targets, one codebase. This document is the
> map: what lives where, how the pieces talk to each other, how each target
> ships, and how the structure scales later. Read this before reorganising
> anything. Last updated: 2026-06 (post folder-restructure).

---

## 1. Top-level layout — each folder is a target or shared code

```
cedarguard/
├── web/            React + Vite frontend (the website / SPA). Was `src/`.
├── api/            Serverless backend — ONE action-dispatch API, used by web AND desktop.
│                   Stays at the repo ROOT (Vercel requires api/ under the deploy root).
├── shared/         Code imported by BOTH web/ and api/ (types, constants, shared config).
├── apps/
│   └── desktop/    Electron main process (.cjs) — packages the SAME web renderer into a macOS app.
└── <root configs>  package.json, tsconfig.json, vite.config.ts, vercel.json, firebase.*, *.md …
                    Repo-level config governs all targets → it belongs at root (single-deploy, no workspaces).
```

**Build outputs (gitignored, NOT source — regenerated by builds, never committed):**
- `dist/` — the Vite **web/renderer** bundle (served by Vercel; loaded by the Electron app).
- `dist-electron/` — the **packaged desktop app** (`.app` / `.dmg`) from electron-builder
  (`directories.output: dist-electron`). Deliberately a *separate* name from `dist/` so the
  desktop packager never overwrites the renderer bundle. Both are in `.gitignore` + `.vercelignore`.

**Why config is at the root (not inside `web/`):** this repo is a single npm package (no
workspaces). One `package.json` / `tsconfig` / `vite.config` governs every target, and Vercel
reads `vercel.json` + serves `api/` from the repo root. This is the standard, production-grade
layout for a single-team, single-deploy multi-target product. (To make each folder
self-contained with its own config you must adopt **workspaces** — see §6.)

## 2. `web/` — feature-first frontend

```
web/
├── main.tsx                 Vite entry (index.html → /web/main.tsx)
├── App.tsx                  Router + app shell
├── features/<domain>/       Per-domain code. Domains:
│     governance · technicalAssurance · risk · compliance · programmes ·
│     projects · reporting · admin · learning · chat
│     • governance/ + technicalAssurance/ have pages/ AND components/ (self-contained)
│     • the other 8 domains have pages/ (their components live in the shared lib below)
├── components/              SHARED component library (used across features):
│     layout shell (Header, Sidebar, Mobile*), PageHeader, PageActions, AuthProvider,
│     RoleGuard, ErrorBoundary, table/ (DynamicTable), validation/, common/, dashboard/,
│     chat/, compliance/, admin/, forms/, historicalReporting/, onboarding/, desktop/, public/
├── pages/                   ONLY Login, Landing, public/ (marketing) — public/auth, not feature-app
├── store/                   Zustand store (useStore.ts) + sub-stores
├── lib/                     api.ts, auth/ (bridges), firebase.ts, utils, exportUtils, chatTransport
├── hooks/  data/  services/  constants/  utils/  types/
```

**Conventions (match `bulletproof-react`):**
- Features do **not** import each other — compose at the app/page level.
- Unidirectional flow: shared (`components/`, `hooks/`, `lib/`) → features → app.
- **No barrel `index.ts`** files (keeps Vite tree-shaking effective).
- A new domain feature is `web/features/<domain>/`. Only genuinely cross-cutting
  components belong in `web/components/`.

## 3. `shared/` — the web↔api boundary

```
shared/
├── constants/roleConstants.ts      role enums (ROLE_STRINGS, PM_LEVELS …)
├── types/historicalReporting.ts    historical-reporting types
├── types/technicalAssurance.ts     TAC entity types
└── lib/composerModels.ts           AI model-catalog config (backend imports it)
```

**Rules:**
- Code used by **both** `web/` and `api/` lives here — never `../../web/...` imported from `api/`.
- Imported via **relative paths with `.js` extensions** — **never a tsconfig path alias.**
  (Vercel's serverless build does **not** resolve tsconfig "Path Mappings"; relative imports
  are the proven mechanism.)
- `riskMetrics.ts` / `validation.ts` / `riskConversion.ts` intentionally stay in `web/lib/`
  (they depend on web-only modules and the backend doesn't import them).

## 4. The API + auth wiring (web AND desktop hit the SAME backend)

```
                         ┌─────────────────────────────┐
  web  (browser) ───────▶│  /api?action=<name>         │
                         │  api/index.ts (dispatcher)  │──▶ Firestore (Admin SDK)
  desktop (Electron) ───▶│  → routes/* → lib/*         │──▶ Gemini / OpenRouter (AI)
                         └─────────────────────────────┘
```

- **One backend.** `api/` is a platform-agnostic Express-style action dispatcher
  (`POST /api?action=foo`). Both clients call the exact same endpoints.
- **API base resolution** (`web/lib/api.ts`):
  - Web → relative `/api` (Vercel rewrites to the serverless function).
  - Desktop → `window.cedar.apiBaseUrl` (the production API URL, injected by the
    Electron main process via preload).
- **Auth bridge** (`web/lib/auth/authBridge.ts`) — selected at module load by `isDesktop`:
  - Web → `firebaseWebBridge` (Firebase Web SDK Google login).
  - Desktop → `desktopIpcBridge` → `apps/desktop/googleOAuth.cjs` (Google OAuth PKCE +
    loopback + Firebase REST `signInWithIdp`; tokens encrypted with `safeStorage`).
  - **Never read `auth.currentUser` or call `signOut(auth)` directly** — always go through
    the bridge (it's a no-op on desktop otherwise).
- **Firebase**: Auth + Firestore. Client init in `web/lib/firebase.ts` (no Storage export).
  Server uses the Admin SDK in `api/lib/context.ts`. Uploads go through
  `api/lib/storage.ts uploadAsset` (base64 → API), never the client Storage SDK.

## 5. Deployment targets

### Web → Vercel (primary)
- `vercel.json` at root; `api/` at root → bundled into **one** serverless function
  (`api/index.ts` is the only handler; `routes/` + `lib/` + `shared/` are traced dependencies).
- SPA rewrite + 4 cron jobs (`runChaseEngine`, `hrcRunMonthlySnapshot`,
  `hrcRunRetentionPurge`, `tacRefreshCorpus`).
- Verify a change before promoting: `vercel deploy` (preview) → check `readyState: READY`.

### Web/API → Azure App Service (secondary, future-ready)
- The **same** `api/` handler runs on Azure App Service — it's not Vercel-specific.
- CI: `.github/workflows/main_cederguard-property-compliance.yml` (build + test → Azure).
- Only Azure-specific piece is platform config (App Service settings / pipeline), **not code** —
  "one backend, platform adapters at the edge."

### Desktop → macOS DMG (Apple notarized)
- `npm run dist:mac` → `scripts/buildDesktopEnv.cjs` (bakes VITE_* into `apps/desktop/built-env.cjs`)
  → `build:desktop-web` (Vite renderer build) → `electron-builder --mac`.
- `electron-builder.yml`: `appId: com.cedarguard.desktop`, `hardenedRuntime: true`,
  `entitlements: build/entitlements.mac.plist`, `notarize: true`.
- Hardened runtime + entitlements + notarization keep the app **Gatekeeper-clean**
  (no install warnings/blocks). Lights up fully once an Apple Developer cert + signing
  env vars are present. CI tag `v*` builds the signed DMG.
- The desktop app **reuses the web renderer** (the built `dist/`), so there is no separate
  frontend codebase to maintain.

## 6. Scaling path — when (and how) to adopt workspaces

The current layout is intentionally **not** a workspaces monorepo. That is correct for a
single team / single product. Adopt workspaces (pnpm/npm) + Turborepo **only** when a
concrete trigger appears:

- a second app starts sharing versioned packages, **or**
- a second team needs independent ownership / dependency isolation, **or**
- CI/build times genuinely hurt.

Doing it speculatively is the documented "premature-monorepo regret" — long-lived,
hard-to-reverse decisions taken before they're needed.

**The migration is mechanical because today's layout maps 1:1 onto the end-state:**

```
cedarguard/
├── apps/
│   ├── web/        (own package.json, tsconfig, vite.config)   ← from web/
│   ├── api/        (own package.json)                          ← from api/
│   └── desktop/    (own package.json)                          ← from apps/desktop/
├── packages/
│   ├── shared/     (types, constants, pure logic)              ← from shared/
│   └── config/     (shared tsconfig/eslint)
├── pnpm-workspace.yaml + turbo.json
```
Note: `api/` must remain reachable at the Vercel deploy root (root-dir setting or a
build-orchestrated output), and the no-path-alias rule still applies to the Vercel function.

## 7. The golden rules (don't break these)

- **Verification gate** for any non-trivial change: `npx tsc --noEmit` **and** `npm run build`
  — both must exit clean. For deploy-affecting changes, also a Vercel **preview** deploy.
- **`api/` stays at the repo root** (Vercel requirement). Don't move it or `apps/desktop/`.
- **Shared code → `shared/`**, relative `.js` imports, **no path alias** (breaks Vercel functions).
- **`api/routes/ai.ts` is out of bounds.** Storage only via `api/lib/storage.ts`.
- **No `firebase/storage` import in `web/`**; never read `auth.currentUser` directly (use the bridge).
- Big files (`useStore.ts`, large `api/routes/*`, `Dashboard.tsx`, `ComplianceSetup.tsx`) are
  known split candidates — a deliberate future task, not a side effect.
