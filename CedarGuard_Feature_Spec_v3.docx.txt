PART 1 — PRODUCT OVERVIEW & SYSTEM ARCHITECTURE
	

1. Product Overview
CedarGuard is an enterprise-grade, cloud-hosted AI-powered Risk and Compliance Management System built for social housing delivery organisations operating in the United Kingdom. The platform is powered by the Cedar Intelligence Engine, which automates identification, tracking, management, and financial quantification of compliance obligations and project risks across the full delivery lifecycle.
The platform is accessed at www.cedarguard.co.uk. All users sign in at the same login page. Upon signing in, each user is taken directly into their own primary working layer — determined by their role. Project Managers are taken into the Project Management layer where they see and manage their assigned projects. Programme Managers are taken into the Programme layer. Portfolio-level users are taken into the Portfolio aggregate view. Every user works within their own layer as their default — this is not a shared starting point. There are no separate sub-applications or separate portals — it is one unified product with role-based layers.


1.1 Core Value Proposition
* AI-driven compliance identification mapped to current UK social housing regulations via the Cedar Intelligence Engine
* Automated risk identification and mitigation strategy generation across 7 structured risk profiling phases
* Financial risk quantification through Annual Loss Expectancy (ALE) — Gross and Residual — across all workstreams
* Risk Appetite Management — alignment tracking (Aligned / Misaligned / Pending) against stated organisational appetite
* Real-time dashboards aggregating data across all projects in a programme
* Full audit trail and evidence repository for regulatory defensibility
* Proactive alerting: Critical, KRI Breaches, and Statutory alert categories with push notification dispatch
* Structured lifecycle management: Risk → Managed/Mitigated/Tolerated/Closed; Issue: Investigating → Escalated → Implementing Fix → Resolved
* Lessons Learned repository with Cost Avoidance (£) tracking and knowledge hub export

2. System Architecture — Role-Based Layers & Active Context
CedarGuard is a single application with a single sign-in. Upon login, the platform detects the user's role and loads their primary layer automatically — a Project Manager is taken into their project context, a Programme Manager into their programme context, a Portfolio user into the portfolio aggregate view. The Active Context switcher is always available in the header and allows users with the appropriate permissions to switch scope — but each user's starting point (their primary layer) is determined by their role, not by manual selection. This means Project Managers are introduced to the platform through the Project Management layer, not through a programme or portfolio view.


2.1 Active Context Switcher
The Active Context dropdown is displayed in the persistent top header bar, always visible regardless of which page the user is on. On login, the platform automatically sets the Active Context to the user's primary layer (project for Project Managers, programme for Programme Managers, portfolio for Portfolio users). The context can then be changed by users with appropriate permissions — changing it shifts the scope of all data displayed across modules (dashboards, registers, trackers, reports, and alerts all respond to the selected context).

* The 'Initiate New Programme' button is persistently displayed in the context bar — accessible from any page
* Context selection persists across navigation — changing page does not reset context
* All data displayed on dashboards, registers, trackers, and reports reflects the active context
* URL parameters (?type=programme) indicate context scope but the context switcher is the primary control


2.2 Hierarchy & Data Flow
The interface is unified — same application, same login. Data is structured in a three-tier hierarchy. Each user's role determines which tier they land in on login (their primary layer). Data flows upward from project to programme to portfolio aggregate — never downward. The Active Context switcher allows authorised users to navigate across layers.

* Risk Register allows filtering between 'Programme Level Only' and 'All Projects' within a programme
* Risk Aggregation Data page provides a dedicated cross-project consolidated view

4. Platform Onboarding & Publication Workflow
Before a programme or project is active and visible in dashboards and reports, it must complete a mandatory 4-step publication workflow. This workflow is enforced by the system — each step must be completed before the next is enabled. The publication gate ensures data quality and AI analysis completeness before any programme or project enters the live delivery environment.


Step
	Action
	Description
	Gate Condition
	1
	Create Project/Programme
	Initial creation with core metadata (name, type, dates, team, RIBA alignment)
	Mandatory fields completed
	2
	Complete Compliance Setup
	Run AI compliance analysis — Cedar Intelligence Engine identifies applicable regulations and populates the Compliance Tracker
	Compliance Setup confirmed by user
	3
	Complete Risk Setup
	Run AI risk identification — Cedar Intelligence Engine identifies risks across 7 profiling phases. Minimum 3 strategic risks must be registered
	≥3 risks confirmed in Risk Register
	4
	Publish
	'PUBLISH PROJECT' button enabled. Programme/Project becomes active and data flows to dashboards, reports, and aggregation views
	Steps 1–3 all completed
	

* The Setup Progress sidebar tracks completion of all 4 steps with visual status indicators
* An 'AI Readiness' indicator shows model readiness before AI analysis is run
* 'Load Demo Strategy Template' and 'Load Demo' buttons are available for onboarding and testing
* 'PUBLISH PROJECT' button remains locked (greyed out) until all 3 preceding steps are complete

	Description
	The Dashboard is the primary landing page. It provides a programme-level aggregated view of compliance and risk status across all projects, with quick access to key actions and AI setup triggers.

	* Active Context (from header switcher)
* Project/programme context selector (within dashboard)
* Compliance Setup status
* Risk Setup status
	Outputs / Results

* Dashboard data reflects the active context selection
	

6.2 Calendar
6.2 — Calendar (/calendar)
	Description
	The Calendar provides a unified timeline view of all scheduled events across the active context: project plan milestones from the Project Plan, compliance due dates from the Compliance Tracker, and risk review dates from the Risk Register. Available to both Programme and Project Managers — scope follows the Active Context.
	Business Rules
	* Calendar scope follows Active Context — programme shows all projects; project shows the selected project only
* Event types: Project Plan Milestones, Compliance Due Dates, Risk Review Dates
* Events are colour-coded by type and by RAG/urgency status — overdue events are highlighted in red
* Clicking an event navigates to the source record (milestone, compliance item, or risk)
* Calendar does not create events — events are managed in their source modules
* View modes: Month, Week, Day, List
* Filter by event type: All Events | Milestones | Compliance | Risk Reviews
	Inputs / Fields
	* Active Context (from header switcher)
* Event type filter (All / Milestones / Compliance / Risk Reviews)
* View mode selector (Month / Week / Day / List)
* Date navigation (forward / back / Today)
	Outputs / Results
	* Calendar grid / list with all in-scope events — colour-coded by type and urgency
* Click-through to source record for each event
* Summary count: events this week, overdue events
* Empty state when no events exist in the selected date range
	Acceptance Criteria
	* Calendar loads within 3 seconds with all events for the active context
* Filter by Milestones returns only milestone events — no compliance or risk events shown
* Overdue events display in red on all view modes
* Clicking a compliance event navigates correctly to the Compliance Tracker item
* Clicking a milestone event navigates correctly to the Project Plan milestone record
	

6.3 My Tasks

	Description
	My Tasks is a personal task board aggregating all action items assigned to the currently logged-in user across the platform — from the Compliance Tracker, Risk Register, and Issues Log. 	Outputs / Results

* Task cards: task title, source type badge (Compliance / Risk / Issue), source reference, due date, status, project name

PART 4 — PROJECT INITIATION MODULE
	

The Project Initiation module provides the two core pages for creating and planning a project: the Project Initiation form (to register and configure the project) and the Project Plan (to schedule RIBA-aligned milestones and key delivery dates). 7. Project Initiation Module
7.1 Project Initiation
7.1 — Project Initiation — Project Creation & Configuration
	Description
	The Project Initiation page is used to create and configure a new project within CedarGuard. It captures all essential project metadata: programme linkage, scheme type, RIBA stage, HRB designation, assigned PM, and key dates. Once created the project enters the 4-step publication workflow before going live.
	Business Rules
	* Every project must be linked to a parent programme — orphan projects are not permitted
* Reference Code is auto-generated by the system (format: PRJ-XXXX) — read-only
* HRB (Higher-Risk Building) flag triggers additional Building Safety Act 2022 compliance requirements via Cedar Intelligence Engine
* RIBA Stage sets the current delivery phase — used by the All Projects filter panel and Project Plan
* Assigned PM must be an existing platform user
* Project cannot be published until all 4 steps in the publication workflow are complete
* A 'Load Demo' button populates the form with sample data for onboarding
* Editing an existing project updates metadata and may trigger re-analysis if HRB flag or Scheme Type changes
	Inputs / Fields
	* Project Name (text, required)
* Reference Code (auto-generated, read-only)
* Parent Programme (dropdown — required)
* Scheme Type: New Build – Higher-Risk Building (HRB) | New Build – General Needs | Retrofit & Decarbonisation | Refurbishment | Estate Regeneration | Damp & Mould Remediation
* RIBA Stage: S0 Strategic Definition | S1 Preparation & Briefing | S2 Concept Design | S3 Spatial Coordination | S4 Technical Design | S5 Manufacturing & Construction | S6 Handover | S7 Use
* Location / Address (text)
* HRB Flag (checkbox: Higher-Risk Building designation)
* Assigned Project Manager (user selector)
* Project Start Date (date picker)
* Expected Completion Date (date picker)
* Project Description / Scope (text area)
* Additional Flags: Brownfield Site | S106 Obligations | Leasehold Units | BIM / Digital Thread | SHDF Funding | PCR 2015 / Procurement Act
	Outputs / Results
	* Project record created and saved
* Project appears in All Projects page under the linked programme
* Project available in Active Context switcher
* 4-step publication workflow initiated
* HRB flag: Building Safety Act compliance requirements queued for Compliance Setup
* Project Plan scaffold created (empty RIBA stage structure)
	Acceptance Criteria
	* Project Name required — form blocks submission without it
* Parent Programme required — form blocks submission without it
* Reference Code is unique across the workspace
* HRB flag change on edit shows confirmation: 'This will update applicable compliance requirements'
* Project appears in All Projects within 60 seconds of creation
* Project available in Active Context switcher immediately after creation
	

7.2 Project Plan — RIBA Plan of Works
7.2 — Project Plan (/project/plan) — RIBA-Aligned Milestone Scheduling
	Description
	The Project Plan provides a RIBA Plan of Works-aligned project schedule structured around all 8 RIBA stages (S0–S7). Users plan, track, and update key milestone dates across all delivery phases. Milestones feed into the Calendar module and are used to track programme delivery progress against the baseline. Supports planned and actual dates to measure performance.
	Business Rules
	* Plan is structured around 8 RIBA stages: S0 Strategic Definition | S1 Preparation & Briefing | S2 Concept Design | S3 Spatial Coordination | S4 Technical Design | S5 Manufacturing & Construction | S6 Handover | S7 Use
* Each RIBA stage has: planned start/end dates, actual start/end dates (updated during delivery), Stage Lead, and Notes
* Users can add unlimited custom milestones within any RIBA stage
* Milestones automatically appear in the Calendar module once created
* Stage status is auto-calculated: Planned | In Progress | Complete | Overdue
* The current RIBA stage set in Project Initiation is highlighted on the plan
* Plan can be exported as PDF or Excel schedule
* RIBA stage completion prompts user to update RIBA Stage in Project Initiation
* Programme Managers can view Project Plans for all projects via context switching
	Inputs / Fields
	* RIBA Stage accordion (expand each stage to add/edit milestones and stage dates)
* Stage-level fields: Planned Start | Planned End | Actual Start | Actual End | Stage Lead (user selector) | Stage Notes
* Milestone fields per milestone: Name (required) | Planned Date (required) | Actual Date | Status (auto-calculated) | Owner | Priority (High / Medium / Low) | Notes
* Add Milestone button (within each RIBA stage)
* Export Plan (PDF / Excel)
* Timeline / Gantt view toggle
	Outputs / Results
	* RIBA-structured plan with all 8 stages as expandable sections
* Milestone list per stage with planned vs actual dates and status badges
* Stage-level progress indicator (% milestones complete in stage)
* Overall plan progress indicator (% of all milestones complete)
* Timeline / Gantt view — visual horizontal representation of all stages and milestones
* Calendar integration — all milestones appear in Calendar module
* Exported plan document (PDF or Excel)
	Acceptance Criteria
	* All 8 RIBA stages display in correct sequence (S0 through S7)
* Milestone added to any stage appears in Calendar on the planned date within 5 minutes
* Milestone status auto-updates to Overdue when planned date passes without Actual Date entered
* Stage status auto-updates to Complete when Actual End Date is entered
* Stage progress % = milestones complete / total milestones in stage × 100
* Export PDF renders all stages and milestones in a correctly formatted printable layout
* Timeline view displays all stages and milestones proportionally on horizontal axis

PART 5 — COMPLIANCE MODULE
	

The Compliance Module manages the full lifecycle of regulatory compliance requirements from AI-driven identification through to evidenced compliance. It is one of the two core AI-powered modules on the platform (alongside Risk Management).


7. Compliance Module
7.1 Compliance Setup (AI-Driven) — v4.0
7.1 — Compliance Setup (/compliance/setup?type=programme) — Cedar Intelligence Engine v4.0
	Description
	The Compliance Setup is the entry point for all compliance management. The user completes a 2-phase structured form. The Cedar Intelligence Engine processes the inputs, maps them against the UK Regulations Library, and generates a tailored set of compliance requirements for the programme or project. This is a mandatory step before the Compliance Tracker and Dashboard are activated.
	Business Rules
	* Compliance Setup is a mandatory step in the 4-step Publication Workflow (Step 2)
* The setup targets either Programme Compliance or Project Compliance — selected in Phase 1
* All generated compliance requirements reference the specific regulation they are derived from
* User must review and confirm AI-generated requirements before the Compliance Tracker is activated
* Re-running setup creates a new version and preserves the previous version in history
* A 'Load Demo Strategy Template' button is available to pre-populate the form with a demonstration dataset
* The AI Context Hint panel reminds users that more detail in Phase 2 produces more accurate AI output
* Current system version: v4.0 (as displayed on the page)
* Two CTAs are available: 'Conduct Detailed Assessment' (questionnaire-driven) and 'Consult CedarGuard AI' (AI-assisted)
	Inputs / Fields
	* PHASE 1 — Context Selection:
* Assessment Target (radio/select: Programme Compliance / Project Compliance)
* Select Registered Programme (dropdown — lists all registered programmes)
   * Once selected programme happens all details should automatically populate the differents fields


* PHASE 2 — Programme Delivery Profile:
* Reference Name (text field, e.g. 'Northeast Regeneration Phase II')
* Assessment Type (dropdown): New Build Development / Retrofit & Decarbonisation / Estate Regeneration / Asset Management & Compliance / Mixed Use Scheme
* Location / Region (text field, e.g. 'Manchester, UK')
* Strategic Objectives / Scope (text area — scale, purpose, key deliverables)
* Technical Characteristics (checkbox group): Higher-Risk (HRB), Brownfield Site, S106 Obligations, Major Demolition, BIM / Digital Thread, PCR 2015 Procurement, SHDF Funding, Leasehold Units
	Conduct Detailed Assessment 
	* Predetermined programme level sets of questions to understand the full context, situation and setup of the programme
	Outputs / Results
	* Compliance requirement list — each record contains: ID, regulation reference, requirement description, due date, responsible owner, status, priority
* Compliance Tracker auto-populated with all generated requirements
* Compliance Dashboard activated with RAG status
* Compliance Alerts configured
* Setup summary available for review
	Acceptance Criteria
	* Cedar Intelligence Engine generates a minimum of 5 compliance requirements for any valid form submission
* Every requirement references at least one regulation from the UK Regulations Library
* Requirements appear in the Compliance Tracker within 60 seconds of AI processing completing
* User can add, edit, or remove AI-generated requirements before confirming
* Re-run creates a new version — previous setup is preserved in history
* Both CTAs ('Conduct Detailed Assessment' and 'Consult CedarGuard AI') are functional from Phase 2

7.3 Compliance Tracker
7.3 — Compliance Tracker (/compliance/tracker?type=programme)
	Description
	The Compliance Tracker is the central operational tool for managing all compliance requirements. It includes embedded AI assistance, multiple view modes, and a full KPI header bar. It tracks every compliance item from identification through to evidenced compliance.
	Business Rules

8.1 Risk Rating Scale
IMPORTANT: The CedarGuard platform uses a 5-tier risk rating scale based on a Probability × Impact (PxI) score of 1–25. This is different from simpler Low/Medium/High/Critical scales. All risk-related features, database fields, UI labels, and reports must use this scale.

* Gross Risk Rating = Probability Score (1–5) × Impact Score (1–5) = 1–25
* Residual Risk Rating = Post-mitigation Probability × Post-mitigation Impact
* Risk Reduction % = ((Gross ALE − Residual ALE) / Gross ALE) × 100
* Severe risks (≥16) are surfaced in the 'High/Severe' KPI on all dashboards

PART 13 — KEY SYSTEM LOGIC
	

12. ALE — Annual Loss Expectancy
Annual Loss Expectancy (ALE) is the primary financial risk quantification metric on the CedarGuard platform. It is calculated and tracked at both Gross (pre-mitigation) and Residual (post-mitigation) levels for every risk record. ALE data appears on the Risk Dashboard, Risk Register, Aggregation page, KRI Tracker, and Executive One-Pager.


Metric
	Formula
	Where Used
	Gross ALE
	Impact (£) × Probability (%)
	Risk Register (per risk), Risk Dashboard (ALE by Workstream chart), Aggregation page
	Residual ALE
	Residual Impact (£) × Residual Probability (%)
	Risk Register, Risk Dashboard KPI, Aggregation page Summary by Project, KRI Tracker (Exp £)
	Risk Reduction %
	((Gross ALE − Residual ALE) / Gross ALE) × 100
	Risk Register (per risk), Risk Dashboard KPI, KRI Tracker (Reduction %)
	Total Portfolio Exposure (£)
	Sum of all Gross ALE values in scope
	Executive One-Pager
	

* ALE fields are displayed in GBP (£) throughout the platform
* Risk Reduction % is auto-calculated — it updates when Gross or Residual ALE values change
* ALE by Workstream chart groups Gross and Residual ALE by the Workstream field on each risk
* Risks without Impact (£) or Probability (%) values entered will show £0 ALE — users should be prompted to complete these fields


13. Risk Appetite Management
Risk Appetite is a first-class feature on the CedarGuard platform. It determines whether individual risks are aligned with or misaligned with the organisation's stated risk appetite. Risk Appetite is set at the programme level and evaluated against each risk's Residual Risk Rating.

16. Cedar Intelligence Engine — AI Pipelines
16.1 Compliance AI Pipeline
Stage
	Description
	1. Input
	2-phase form data: Assessment Type, Location, Scope, Technical Characteristics (HRB, Brownfield, S106, BIM, PCR 2015, SHDF, Leasehold, Demolition)
	2. Pre-processing
	Normalise inputs. Map Assessment Type to applicable regulatory domains. Map HRB flag to Building Safety Act threshold conditions.
	3. Regulation Mapping
	Match normalised inputs to Regulations Library using rules-based + semantic matching. Each match produces: regulation ID, applicability score, mandatory flag.
	4. Requirement Generation
	For each matched regulation, generate a compliance requirement: title, description, regulation reference, due date formula, default priority, evidence type required.
	5. Deduplication
	Remove duplicate requirements (same regulation + same due date formula). Merge where applicable.
	6. Output
	Ordered compliance requirement list for user review. Inserted into Compliance Tracker on confirmation.
	7. Audit Trail
	Input payload, model version, regulation mapping results, and output records logged to AI processing audit log.
	

16.2 Risk AI Pipeline
Stage
	Description
	1. Input
	7-phase questionnaire responses (30+ questions across governance, supply chain, site, construction, residents, dependencies, data readiness)
	2. Context Enrichment
	Load Lessons Learned entries matching programme type, RIBA stage, and risk categories as contextual signals.
	3. Risk Identification
	Apply risk taxonomy across 12 categories and 12 workstreams to identify likely risks for the given programme context.
	4. Scoring
	Assign initial Gross Probability (1–5) and Impact (1–5) scores based on questionnaire responses and historical patterns.
	5. ALE Initialisation
	Set initial Gross Impact (£) estimate and Probability (%) to initialise Gross ALE calculation.
	6. Mitigation Generation
	For each identified risk, generate initial mitigation strategy with suggested controls and owner role.
	7. Deduplication
	Detect and merge conceptually duplicate risks before output.
	8. Output
	Ordered risk list (by Gross Rating, highest first) for user review. Inserted into Risk Register on confirmation.
	9. Audit Trail
	Input payload, model version, scoring rationale, and output records logged to AI processing audit log.
